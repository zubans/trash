# Сервис авторизации (`AuthService`)

## Назначение

`AuthService` отвечает за первый этап жизненного цикла пользователя:

- регистрацию нового пользователя по номеру телефона и паролю;
- аутентификацию существующего пользователя;
- выпуск подписанного JWT-токена после успешной аутентификации.

Сервис не занимается HTTP-протоколом и не ходит напрямую в таблицы базы данных. Он принимает на вход абстракцию `repository.UserRepository`, что позволяет тестировать бизнес-логику отдельно от слоя хранения.

## Расположение в коде

```
backend/
├── service/
│   ├── auth.go       # реализация AuthService
│   └── auth_test.go  # юнит-тесты сервиса
├── repository/
│   └── user.go       # контракт UserRepository и реализация на SQL
└── handler.go        # HTTP-обработчики, вызывающие AuthService
```

## Зависимости

Сервис зависит только от интерфейса `repository.UserRepository`:

```go
type UserRepository interface {
    FindByPhone(phone string) (*User, error)
    Create(user *User) error
}
```

В качестве второй зависимости выступает секретный ключ для подписи JWT. Он передаётся при создании сервиса.

## Создание сервиса

### Из переменных окружения

```go
repo := repository.New(db)
authSvc := service.NewAuthService(repo)
```

Функция читает переменную окружения `JWT_SECRET`. Если она не задана, используется development-значение `dev-secret-change-me`.

### С явным секретом

```go
authSvc := service.NewAuthServiceWithSecret(repo, "very-strong-secret")
```

Используется в тестах и в окружениях, где секрет передаётся явно, а не через переменные окружения.

## Публичные методы

### Register

```go
func (s *AuthService) Register(phone, password string) (*repository.User, error)
```

Создаёт нового пользователя.

**Поведение:**
1. Проверяет, что `phone` и `password` не пустые. В противном случае возвращает ошибку `phone and password are required`.
2. Вызывает `FindByPhone`. Если пользователь найден, возвращает ошибку `user already exists`.
3. Если `FindByPhone` вернул ошибку, отличную от `sql.ErrNoRows`, пробрасывает её вызывающему коду.
4. Хеширует пароль алгоритмом `bcrypt` со стандартной стоимостью (`bcrypt.DefaultCost`).
5. Создаёт запись пользователя с полями:
   - `Role`: `CUSTOMER`;
   - `Status`: `ACTIVE`;
   - `Balance`: `0`;
   - `Phone`: переданный номер телефона;
   - `Password`: bcrypt-хеш.
6. Повторно вызывает `FindByPhone`, чтобы вернуть созданного пользователя.

**Возвращаемые ошибки:**
- `phone and password are required` — пустой телефон или пароль.
- `user already exists` — пользователь с таким телефоном уже зарегистрирован.
- Ошибки хеширования или репозитория — пробрасываются как есть.

### Authenticate

```go
func (s *AuthService) Authenticate(phone, password string) (*repository.User, error)
```

Проверяет учётные данные пользователя.

**Поведение:**
1. Проверяет, что `phone` и `password` не пустые.
2. Ищет пользователя по телефону через `FindByPhone`.
3. Сравнивает переданный пароль с сохранённым bcrypt-хешом.
4. При успешном сравнении возвращает пользователя.

**Возвращаемые ошибки:**
- `phone and password are required` — пустой телефон или пароль.
- `invalid credentials` — пользователь не найден, пароль неверный или репозиторий вернул ошибку.

Сообщение `invalid credentials` используется как единый ответ на любую проблему с учётными данными, чтобы не давать злоумышленнику информации о существовании аккаунта.

### GenerateJWT

```go
func (s *AuthService) GenerateJWT(user *repository.User) (string, error)
```

Создаёт подписанный JWT-токен для аутентифицированного пользователя.

**Поведение:**
1. Формирует токен алгоритмом `HS256`.
2. Записывает в claims:
   - `sub` — UUID пользователя в строковом виде;
   - `phone` — номер телефона;
   - `role` — роль пользователя;
   - `exp` — время истечения (`now + 24 часов`).
3. Подписывает токен секретным ключом сервиса.

**Возвращаемые ошибки:**
- Ошибки подписи токена пробрасываются из библиотеки `golang-jwt/jwt/v5`.

## Безопасность

### Хранение паролей

- Пароль никогда не сохраняется в открытом виде.
- Используется `bcrypt` с параметром `bcrypt.DefaultCost`.
- Хеширование выполняется в слое сервиса, а не в репозитории. Репозиторий получает уже готовый хеш и записывает его в колонку `password`.

### JWT

- Алгоритм подписи: `HS256` (HMAC + SHA-256).
- Время жизни токена: 24 часа.
- Секретный ключ задаётся через переменную окружения `JWT_SECRET`.
- **В продакшене обязательно задайте сильный и уникальный `JWT_SECRET`. Development-значение не должно использоваться в продакшене.**

### Сокрытие информации об ошибках

`Authenticate` возвращает одинаковое сообщение `invalid credentials` при отсутствии пользователя, неверном пароле и внутренней ошибке репозитория. Это затрудняет перебор номеров телефонов.

## Переменные окружения

| Переменная | Назначение | Значение по умолчанию |
|------------|------------|----------------------|
| `JWT_SECRET` | Секретный ключ для подписи JWT | `dev-secret-change-me` |

## Пример использования

```go
db, err := sql.Open("postgres", dsn)
if err != nil {
    log.Fatal(err)
}

userRepo := repository.New(db)
authSvc := service.NewAuthService(userRepo)

// Регистрация
user, err := authSvc.Register("+79001234567", "secret-password")
if err != nil {
    log.Fatal(err)
}

// Аутентификация
user, err = authSvc.Authenticate("+79001234567", "secret-password")
if err != nil {
    log.Fatal(err)
}

// Получение токена
token, err := authSvc.GenerateJWT(user)
if err != nil {
    log.Fatal(err)
}
```

## Тестирование

Юнит-тесты расположены в `backend/service/auth_test.go` и используют in-memory мок репозитория. Покрываются следующие сценарии:

- успешная регистрация с проверкой хеша пароля;
- регистрация с пустым телефоном или паролем;
- регистрация дублирующего пользователя;
- ошибки репозитория на этапе поиска и создания;
- успешная аутентификация;
- аутентификация с пустыми полями;
- аутентификация несуществующего пользователя;
- аутентификация с неверным паролем;
- ошибка репозитория при аутентификации;
- генерация JWT и проверка claims;
- проверка, что токен невалиден при проверке другим секретом.

Запуск тестов:

```bash
cd backend
go test ./service/... -v
```

## Ограничения текущей реализации

- Нет SMS-верификации номера телефона.
- Нет проверки формата телефона (допускается любая непустая строка).
- Нет требований к сложности пароля.
- При регистрации не создаётся запись в `customer_profiles` или `executor_profiles`.
- Нет механизма обновления или сброса пароля.
- Нет механизма отзыва (revocation) токенов.
- Нет refresh-токенов.
